Starting with an nmap scan using a quick verbose scan to save time.

![Nmap Scan Placeholder](path/to/screenshot1.png)

The scan reveals some open ports, including 22, 80, and 2222. Let's dive deeper into these ports with a more focused scan.

![Port Scan Placeholder](path/to/screenshot2.png) ![Port Scan Results Placeholder](path/to/screenshot3.png)

Visiting the page, we find a domain: itrc.ssg.htb. We add it to our /etc/hosts file. This discovery suggests the potential for multiple subdomains or Vhosts, so let’s start fuzzing to uncover them.

![Fuzzing Subdomains Placeholder](path/to/screenshot4.png)

Using the n0kovo_subdomains.txt wordlist, we identify a new Vhost: signserv.ssg.htb, which is added to our hosts file.

Analyzing the Webpage

Accessing port 80, we are greeted by an IT support center page.

![IT Support Center Placeholder](path/to/screenshot5.png)

Following our standard methodology, we check the source code but find nothing useful. While running a gobuster scan in the background, we register an account on the homepage and log in.

![Gobuster Scan Placeholder](path/to/screenshot6.png) ![Login Placeholder](path/to/screenshot7.png)

Identifying Potential Vulnerabilities

Upon logging in, we notice a page parameter that could be vulnerable to LFI attacks. For now, we’ll note it down for later testing. We are also presented with a dashboard.

![Dashboard Placeholder](path/to/screenshot8.png)

Next, we explore the file upload feature by creating a new ticket where we can upload ZIP files.

![File Upload Placeholder](path/to/screenshot9.png)

Testing the upload with an arbitrary file, we receive some interesting findings from our directory scan.

![Directory Scan Placeholder](path/to/screenshot10.png)

Exploiting the LFI Vulnerability

Returning to our earlier observation, we test the page parameter with the findings from gobuster. Although admin.php didn't work initially, removing the extension brings us to a new page.

![LFI Parameter Placeholder](path/to/screenshot11.png)

Testing with the full path discovered, we confirm the LFI vulnerability. Now, we prepare a PHP web shell file, compress it into a ZIP file, and upload it.

![PHP Web Shell Placeholder](path/to/screenshot12.png)

Achieving Remote Code Execution

Once the ZIP file is uploaded, we use the phar:// wrapper to access the shell.

![Accessing Shell Placeholder](path/to/screenshot13.png)

The payload works as expected, and now it’s time to catch a reverse shell by starting a listener and using a URL-encoded payload.

![Reverse Shell Placeholder](path/to/screenshot14.png)

Escalating Privileges

Exploring /var/www/itrc, we discover a db.php file with credentials. Unfortunately, a password spray attempt fails. However, in the /uploads directory, we find multiple ZIP files.

![ZIP Files Placeholder](path/to/screenshot15.png)

Extracting and analyzing the uncommon files leads us to the next stage of our attack.

![File Extraction Placeholder](path/to/screenshot16.png)

Compromising the SSH Service

After obtaining a certificate authority private key, we generate an SSH key pair and sign it with the CA. This allows us to SSH into the server as the root user.

![SSH Key Pair Placeholder](path/to/screenshot17.png) ![SSH Login Placeholder](path/to/screenshot18.png)

Exploiting Script Vulnerabilities

On further inspection, we find a Bash script vulnerable to command injection. Using this vulnerability, we leak the CA key and sign a new SSH key pair to gain access to the system as the root user.

![Script Exploit Placeholder](path/to/screenshot19.png)

Finally, we SSH into the system with our newly signed key, gaining full access to the target.

![Root Access Placeholder](path/to/screenshot20.png)

This post highlights the importance of thorough reconnaissance, testing for common vulnerabilities, and leveraging discovered information to escalate privileges and gain full access to a target system. Stay tuned for more writeups and insights into the world of pentesting.
